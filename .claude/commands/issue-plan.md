# 이슈 우선순위 분석

다음 순서로 분석하고, 마지막에 추천 개발 순서를 제안한다.

## 1. 이슈 목록 수집

```bash
gh issue list --state open --limit 50
```

각 이슈의 상세 내용도 확인한다:

```bash
gh issue view <번호>
```

## 2. 영향 범위 분류

각 이슈를 아래 기준으로 분류한다:

- **DB 변경** — `portfolio_assets`, `execution_records` 등 테이블 스키마 영향 여부
- **공유 패키지 변경** — `packages/lib` 타입·스키마, `packages/ui` 컴포넌트 변경 여부
- **앱 전용 변경** — `apps/web` 내부에만 영향
- **의존 관계** — 다른 이슈가 먼저 완료돼야 착수 가능한지

## 3. 우선순위 판단 기준

1. 다른 이슈가 의존하는 이슈 먼저 (블로킹 이슈)
2. DB·공유 패키지 변경이 필요한 이슈 먼저 (범위가 넓을수록 선행)
3. 독립적이고 범위가 좁은 이슈 마지막

## 4. 결과 출력 형식

아래 형식으로 정리한다:

```
추천 개발 순서:
1. #N 이슈제목 — 이유 (영향 범위: ...)
2. #N 이슈제목 — 이유 (영향 범위: ...)
...

주의사항:
- 병렬 진행 가능한 이슈: ...
- 선행 완료 후 착수 가능한 이슈: ...
```
